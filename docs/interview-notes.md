# 面试讲解要点

## 30 秒版本

我学食品科学，做这个项目不是为了套一个通用聊天机器人，而是想验证食品企业标签合规初筛能不能被 AI 产品化。食安智问先把问题检索到具体法规条款，再用显式规则给风险等级和整改建议；如果没有依据就拒答。当前 MVP 完全离线，支持 CLI 和静态 Web Demo，已经接入 30 条黄金问题做回归测试。

## 2 分钟版本

这个项目的场景是预包装食品标签合规。企业在审标签时经常要查食品安全法、GB 7718、GB 28050、GB 2760 等标准，比如生产日期、配料表、营养声称、添加剂通用名称、执行标准代号这些问题。

我没有一开始就上大模型，因为合规问答最重要的不是语言流畅，而是三个点：第一，有没有依据；第二，风险判断能不能复现；第三，没有依据时能不能拒答。所以我先做了一个规则 + 检索的 MVP。

数据层面，我整理了 30 条结构化法规条款，包括条款号、主题、适用范围、生效日期、关键词、风险类型、风险等级和来源链接；同时设计了 30 条黄金测试问题，覆盖高风险、需核验、提示三类场景。

系统流程是：用户输入问题，系统按关键词、主题和法规标题检索条款；命中后根据条款风险等级和问题中的高危词做风险判断；最后输出结论、整改建议和引用依据。如果问题不在知识库里，比如问冷链车压缩机，就返回待人工核验。

展示层我没有用 Streamlit，因为免费托管会休眠，面试现场不稳定。我做了静态 Web Demo，可以直接部署到 GitHub Pages，也保留了 Python CLI。现在自动化测试覆盖核心样例和 30 条黄金问题，能证明这个闭环是可复现的。

后续如果继续做，我会把它升级成 BM25 + 向量混合检索，加引用一致性校验、法规版本管理、人工复核队列，再把 LLM 放在表达层，而不是让 LLM 凭记忆判断法规。

## 为什么不是“套壳 RAG”

- 合规场景的核心不是生成，而是证据、版本、适用范围和责任边界。
- 条款原文与版本元数据是事实层，风险规则是决策层，LLM 只能做表达层。
- “未检索到依据”是正常输出，不是失败；系统必须能拒答。
- 黄金测试集比聊天体验更重要，因为它能复现检索和风险判断。

## 技术取舍

- 先用无依赖关键词检索：小语料下更可控，面试演示更稳定。
- 先不用向量库：当前只有 30 条条款，向量库的工程复杂度大于收益。
- 静态 Web 优先：GitHub Pages 不休眠，适合秋招作品集。
- 风险判断用规则：合规场景不能让模型自由判断高风险或低风险。

## 主动承认边界

目前不是正式法规数据库，不能直接用于企业合规决策。真实落地需要：

- 标准全文授权或官方公开语料。
- 法规生效日期、过渡期、废止/替代关系管理。
- 产品类别路由。
- 标签 OCR 或结构化标签解析。
- 专家标注测试集。
- 人工最终复核和签字。

## 面试官可能追问

**为什么不用 Streamlit？**

Streamlit 适合快速内部原型，但免费托管会休眠，面试现场不稳定。这个项目的前端只是展示和交互，不需要后端，所以静态 Web + GitHub Pages 更稳。

**为什么不用大模型直接回答？**

法规合规不允许模型凭记忆给结论。大模型可以做表达层，但结论必须来自检索到的条款和规则。

**这个项目和食品专业有什么关系？**

问题范围、条款 schema、风险类型和整改建议都来自食品标签合规场景，不是通用文档问答。食品专业背景让我能把“法规条文”转成企业质量部门能使用的审版流程。

**如果企业真实落地怎么做？**

先接入授权法规库和企业内部标准；再做产品类别路由、标签 OCR、规则引擎、人工复核队列和审计日志；LLM 只负责基于证据生成可读报告。
